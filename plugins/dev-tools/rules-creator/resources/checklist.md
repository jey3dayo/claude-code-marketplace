# Rules Quality Checklist

## Settings Rules

- [ ] 明確で曖昧さのない制約記述
- [ ] ルールが存在する理由の説明
- [ ] スコープを明確に定義（適用範囲）
- [ ] 正しい例を提供
- [ ] 不正な例を提供
- [ ] 例外を文書化（存在する場合）
- [ ] 強制メカニズムを指定
- [ ] レビュースケジュールを設定

## Steering Documents

- [ ] 概要が明確なコンテキストを提供
- [ ] 主要決定事項を根拠と共に文書化
- [ ] パターンと規則を明確に説明
- [ ] 技術スタックを文書化
- [ ] AIへのコンテキストがアクション可能
- [ ] よくある落とし穴を識別
- [ ] 例への参照を提供
- [ ] 所有権と更新日を指定

## Hookify Rules

- [ ] トリガーイベントを明確に定義
- [ ] 条件ロジックが明確でテスト可能
- [ ] アクションステップが具体的で実行可能
- [ ] 期待される結果を文書化
- [ ] 失敗処理を指定
- [ ] 設定パラメータをリスト化
- [ ] テスト手順を提供
- [ ] ロギング戦略を定義

## 全般的な品質

- [ ] 既存ルールと競合しない
- [ ] プロジェクト哲学と一致
- [ ] 測定可能/検証可能
- [ ] 例が実際のプロジェクトパターンを使用
- [ ] 定期的なレビューと更新
- [ ] 明確な所有権を確立

## 一貫性

- [ ] 命名規則に従う（kebab-case）
- [ ] 正しいディレクトリに配置
- [ ] 確立されたテンプレート構造を使用
- [ ] 他のルールと一貫した用語
- [ ] 関連ルールを相互参照

## テスト可能性

- [ ] ルール違反が検出可能
- [ ] 強制メカニズムが動作
- [ ] フィードバックが明確でアクション可能
- [ ] エッジケースを考慮
