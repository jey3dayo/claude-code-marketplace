# Rules Creator

効果的なルール、steering、hookifyを作成するためのガイドです。

## このスキルについて

Rulesは、Claude AIの動作を制御し、プロジェクト標準を強制するための仕組みです。4つの異なるタイプがあり、それぞれ異なる目的と強制レベルを持ちます。

### スキルが提供するもの

1. **4つのルールタイプの理解** - Guidelines/Rules/Steering/Hookify
2. **タイプ選択の判断基準** - いつどれを使うべきか
3. **テンプレートとチェックリスト** - 実用的なスターターキット
4. **実例** - 既存ルールから抽出したベストプラクティス

## ルールの階層構造

### 4つのルールタイプ

| タイプ         | 強制レベル | 場所                        | 目的                         |
| -------------- | ---------- | --------------------------- | ---------------------------- |
| **Guidelines** | 提案       | CLAUDE.md                   | 開発哲学、ベストプラクティス |
| **Rules**      | 要求       | .kiro/settings/rules/       | 強制可能な標準、制約         |
| **Steering**   | 参照       | .kiro/steering/             | プロジェクトメモリ、パターン |
| **Hookify**    | 自動化     | .claude/hookify.\*.local.md | イベント駆動の自動アクション |

## タイプ選択の判断基準

### いつGuidelinesを使うべきか

**使用すべき時:**

- 一般的な原則を共有
- 哲学や価値観を伝える
- ベストプラクティスを推奨
- 強制は不要

**例:**

- "型安全性を優先すべき"
- "テストは重要"
- "コードレビューを推奨"

**配置:** CLAUDE.md（プロジェクトルートまたは `~/.claude/`）

### いつSettings Rulesを使うべきか

**使用すべき時:**

- 強制可能な標準が必要
- 明確な制約を定義
- 違反が検出可能
- プロセスルールの文書化

**例:**

- "any型の使用禁止"
- "内部層は外部層をインポート不可"
- "すべてのAPI呼び出しにはエラーハンドリング必須"

**配置:** `.kiro/settings/rules/`

### いつSteeringを使うべきか

**使用すべき時:**

- プロジェクト固有のコンテキスト
- アーキテクチャパターンの記録
- 技術的決定の文書化
- AIのためのプロジェクトメモリ

**例:**

- "Clean Architecture採用"
- "PascalCase for コンポーネント名"
- "相対インポートではなく絶対インポート使用"

**配置:** `.kiro/steering/`

### いつHookifyを使うべきか

**使用すべき時:**

- イベント駆動の自動化
- 危険なコマンドの防止
- Pre/Post操作のフック
- リアルタイム検証

**例:**

- "git push前にテスト実行"
- "rm -rfコマンドをブロック"
- "console.logに警告"

**配置:** `.claude/hookify.*.local.md`
